前言

将 JavaScript 引入网页,首先要解决的就是它和网页的主导语言 HTML 的兼容性问题。在 JavaScript 早期，Netscape 的大佬们希望将 JavaScript 引入 HTML 的同时，不会导致页面渲染出现问题，通过反复的试验和讨论，他们最终达成了向网页中引入通用脚本能力的共识。以现在的眼光看，当初他们很多成果都以 HTML 规范的形式保留了下来。

<script>标签

<script>标签可谓是前端人家喻户晓的东西了，将 JavaScript 插入 HTML 的主要方式就是使用<script>标签，即使在插件、框架天花乱坠的今天，你的 JS 代码经过编译后也会成为一个个呈现在 HTML 的<script>标签，所以不要小瞧它 💪。

属性

<script>这个元素是由 Netscape 创造出来的，最早的 Netscape Navigator 2 中实现，后被正式纳入了 HTML 规范。它有下列 8 个属性（来看看你知道几个 🥴）：

  • src（可选）：表示包含要执行的外部文件。
  • type（可选）：表示 script 标签内容的类型（MIME 类型），代替原来的language属性。按照惯例这个值始终都是text/javascript,但text/javascript和text/ecmascript都已经废弃了 🤡。现在 JavaScript 文件的 MIME 类型通常为application/x-javascript，不过这个值可能导致脚本解析失效，因为在非 IE 浏览器中还有application/javascript和application/ecmascript两个有效值。如果你想使用 ES6 的模块化特性，需要将这个值设置为module后才能使用import和export关键字，不然就会报错 💀。
  • async（可选）：表示立即开始下载脚本，但是不阻塞其他内容的加载、执行，只对外部文件有效。
  • defer（可选）：表示脚本可以延迟到文档完全被解析显示后再执行，只对外部文件有效。
  • charset（可选）：设置src属性指定代码的字符集，但多数浏览器都不在乎它的值，所以很少使用。
  • crossorigin（可选）：配置请求的跨域资源共享（CORS）设置。默认情况不使用 CORS，当这个属性值为anonymous时，配置文件请求不必设置 token 标记；当这个属性值为use-credentials时，配置文件请求必须设置 token 标记。
  • integrity（可选）：允许比对接收资源和加密签名来验证资源的完整性（SRI），如果接受资源的签名和配置的签名不匹配，页面就会报错，脚本不执行，可用于确保内容分发网络（CDN）不提供恶意内容。
  • language（废弃 ⚠️）：表明代码块中的脚本语言类型，如JavaScript,JavaScript 1.2,VBScript等等，但是大多数浏览器都忽略了这个属性，所以世界上本有路，走的人少了，便废弃掉了 😑。

怎么去用？

<script>标签的使用方式有两种，一种是直接在在网页中嵌入 JS 代码，一种是通过 src 属性引入外部的 JS 文件。

<!-- 直接嵌入法 -->
<script>
  console.log("Hello World");
</script>

<!-- 外部引入法 -->
<script src="./your/path/hello_world.js"></script>

内部嵌入

嵌入行内 JS 代码直接将相关内容放在<script>标签中就行了，代码会由上至下的执行，执行的时候会阻塞掉其他操作。

<script>
  console.log("Hello World 1");
  console.log("Hello World 2");
  console.log("Hello World 3");
  console.log("Hello World 4");
</script>

外部引用

要使用外部的 JS 文件，就需要使用src属性，这个属性的值是指向 JS 文件的地址，可以是本地地址也可以是网络地址，执行规则和内嵌保持一致。

<script src="./helloworld.js"></script>

由于<script>是通过 src属性来引入内容的，因此<script>和<img>有着一些相似的特性——<script>可以引入其他域下面的 JS 内容。这份能力允许我们通过不同的域来分发 JS 代码（👉CDN），能让我们将一部分资源、代码放在远端服务器上，用网络带宽来换项目体积；跨域带来便利的同时也带来了安全问题，如果原有的 JS 代码被替换成了恶意代码，用户拉取下来再一执行，直接一个 P0 安全事故按你头上 😵。虽然有integrity这个属性来小校验一波，但是不是每个浏览器都支持，所以安全的事还是给后端背吧 🤤。

还有一点需要提一下，浏览器在解析下面这个资源的时候，会先发送一个 GET 请求取得相应资源。这个初始请求不受同源策略的限制，但返回并执行的 JavaScript 代码会受限制，且这个请求仍然会受到父级页面 HTTP/HTTPS 的限制。

<script src="http://www.baidu.com/core.js"></script>

在哪去用？

在过去，所有<script>都放在页面的<head>中，就像下面代码所示，其目的是为了集中管理 CSS 和 JavaScript。

<!DOCTYPE html>
<html lang="en">
  <head>
    <link href="index.css" />
    <script src="./index.js"></script>
  </head>
  <body></body>
</html>

在怎么去用？的环节我们了解到了 JavaScript 执行会堵塞其他操作，那么把<script>放在<head>无疑会等 JS 加载执行完成后才会开始渲染页面内容（<body>的内容），此期间必然会出现白屏的真空期。

为了解决这个问题，现代 Web 应用程序用了一个投机取巧法，将所有 JavaScript 引用放在<body>元素内容之后，这样页面就会在处理 JavaScript 代码之前渲染页面。白屏时间变短，用户体感 plus plus。

<!DOCTYPE html>
<html lang="en">
  <head>
    <link href="index.css" />
  </head>
  <body>
    <div>...Web Content...</div>
    <script src="./index.js"></script>
  </body>
</html>

defer 与 async

推迟执行脚本

⚠️ 注意：只对外部引用的文件有效嗷

HTML4.01的时候为<script>新增了一个属性defer，这个属性本义表示脚本在执行的时候不会改变页面结构，这意味着脚本会在页面全部解析完成后在运行，再换句话说 🤕（不要打我）就是在表示 HTML 结束</html>标签之后才会执行。这里要小心一点defer的陷阱，这个属性出现相当于告诉浏览器立即下载，延迟执行！

HTML5 规范要求脚本应当按照它们出现的顺序执行，因此控制台会首先打印出 2，因为 1 和 3 的执行已经推迟到 HTML 加载完后，然后按照出现顺序依次打印出 1，3。

/***
 * @File = test1;
 */
console.log(1);

/***
 * @File = test2;
 */
console.log(2);

/***
 * @File = test3;
 */
console.log(3);

<!DOCTYPE html>
<html lang="en">
  <head>
    <link href="index.css" />
  </head>
  <body>
    <div>Hello World</div>
    <script defer src="./test1.js"></script>
    <script src="./test2.js"></script>
    <script defer src="./test3.js"></script>
  </body>
</html>

异步执行脚本

⚠️ 注意：只对外部引用的文件有效嗷

HTML5 中为<script>标签添加了async属性。async和defer相似，它们都只适用于引用外部文件；都改变了脚本的加载方式。不同的是，async的目的是告诉浏览器不必等待当前脚本加载和执行完后再执行加载其他内容，而且它不能保证脚本执行时的顺序，比如两个带有async的script标签，很有可能后者早于前者执行。async的用法和defer，这里就不做代码展示了 😋。

注意事项

代码中慎用</script>

在使用<script>标签的时候需要注意，代码中不能出现</script>标签 ⚠️。因为浏览器在解析行内脚本时，解析到</script>的字样就会把整个代码块当作结束处理，因而导致报错。如果确实需要使用</script>的字样，那么搭配转义符\就完美解决问题了，来梭哈一把下面的代码：

<!-- Error -->
<script>
  console.log('Hello World 1');
  console.log('Hello World 2 </script>');
</script>

<!-- Success -->
<script>
  console.log('Hello World 3');
  console.log('Hello World 4 <\/script>');
</script>

引入外部 JS 不要提前闭口

在日常编辑 HTML 的过程中，不免会出现<img />和<a />、<div />这种提前闭口的单标签。但是在 H5 里这种写法<script />是错误的，这样写浏览器会将其解析为一个 HTML，部分浏览器甚至不能正常处理导致报错。

引入外部 JS 不要在其内部再填充代码

简单来讲，外部引用的 JS 优先级比行内 JS 高。如果一个<script>标签已经引用了外部 JS 文件，还在内部编写代码，那么那部分行内代码就会被忽略掉，浏览器只会下载执行引用的代码。

<noscript>标签

针对早期浏览器不支持 JavaScript 的问题，需要一个优雅的降级方案，最终<noscript>元素诞生了。<noscript>被用于给不支持 JavaScript 的浏览器提供替代内容，这个元素可以包含出<script>以外的任何 HTML 元素。如果浏览器不支持 JS，或者浏览器对 JS 的支持被关闭，这时候会显示<noscript>的内容。

当代的浏览器已经 100%支持 JS 了，但对于禁用 JS 的浏览器来说，这个标签仍能派上用场。下面有一个例子，如果你使用的浏览器支持 JS，你将永远不会看见它；反之就会显示。

<!DOCTYPE html>
<html>
  <head>
    <title>page</title>
  </head>
  <body>
    <script src="./test1.js"></script>
    <script src="./test2.js"></script>
    <noscript>
      <p>还用不支持JS的浏览器，你tnd真是个人才</p>
    </noscript>
  </body>
</html>

文档模式

IE5.5 发明了文档模式的概念，可以通过 HTML 顶部的DOCTYPE来指定文档模式，最初那版有混杂模式（quirks mode）、标准模式（standards mode）。后随着浏览器的普遍实现，又出现了第三种文档模式——准标准模式（almost standards mode）。

混杂模式：混杂模式可以让 IE 浏览器像 IE5 一样支持一些非标准的特性、行为。

标准模式：标准模式让 IE 浏览器具有兼容标准的行为。

准标准模式：准标准模式下浏览器支持很多标准的特性，但是没有标准规定那么严格。

三者区别：标准模式和混杂模式的区别主要体现在 CSS 渲染方面上，还有一些 JS 的副作用场景；准标准模式相比前两者，主要区别在于如何对待图片元素周围的空白。标准模式和准标准模式非常接近，很少需要区分，而且对文档模式的检测也不会对两者进行区分，所以人们说的标准模式很有可能指的是二者中的一个。

小结

  • JavaScript 是通过<script>标签将其插入 HTML 中的，JS 代码可以直接写在<script>标签中作为行内 JS，也可以通过src属性从外部引用作为外部 JS。
  • 要使用外部的 JS 文件，必须使用src属性来指定包含相关资源的URL，而 JS 文件可以和网页在同一台服务器上，也可以在完全不同的域上。
  • 所有的<script>元素会按照它们在网页中的次序依次被解释，因此在不使用async和defer属性的情况下，标签中的代码必须严格按照次序解释，且必须执行完<script>标签中的内容后才会执行后面的内容，又又又因此需要把<script>的内容放在<body>的末尾解释执行。
  • defer属性会把 JS 推迟到整个 HTML 解析渲染完成后在执行，async属性则是告诉浏览器不必等这个脚本加载执行了，直接去加载后面的内容吧。延时脚本能保证后续的执行次序，异步脚本则不能保证执行次序，且这两个属性都只能对外部引用的文件使用。
  • <noscript>可以编写除<script>以外的任何内容。通过使用<noscript>元素，我们可以指定浏览器在不支持 JS 或者支持 JS 的模块被关闭的时候呈现什么内容；反之如果浏览器支持 JS，这些内容不会被渲染。
  • 通过设置DOCTYPE，我们可以指定当前页面使用文档模式的类别。目前存在三种文档模式，混杂模式、标准模式、准标准模式。三者的差异主要体现在 CSS 的渲染，一部分 JS 的副作用，还有图片周围的空白处理方式上。
