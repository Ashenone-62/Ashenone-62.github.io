「散文」如何工程化你的代码

Words count 6.2k Reading time 6 mins. Read count 0

  • 工程化

前言

团队成员代码风格不统一 💩 、BAD CODE被部署上线 💩 、团队编码规范难以落地 💩 、代码质量参差不齐 💩 ，这些问题是协同开发中经常出现的痛点。这篇文章会一步一步教你的配置ESLint，Prettier，Husky，Lint-Staged (有手就会 🙌)。这些工具会在你开发的时候帮你规避掉一些痛点，同时让你慢慢养成代码”洁癖”😊

流程

  • git add .
  • git commit -m "xxxxxx"
  • husky => pre-commit => lint-staged
  • lint-staged => Prettier/ESLint
    • 格式化代码，规范过关 => Success
    • 格式化代码，规范不过关 => Error => 修改代码直到符合规范

保姆级教程，有手就行 🙌

准备工具

框架：React

工具库：ESLint，Prettier，Husky，Lint-Staged

目的：

  • 在你提交代码前通过扫描你的代码发现低级错误和潜在错误，必须修改正确后才能推送到仓库
  • 在提交前会格式化代码，规范化单引号，双引号以及分号的使用

创建项目

# 创建项目
create-react-app yourprojectname
# 进入项目目录
cd yourprojectname

ESLint

# 初始化你的 eslintrc.js
./node_modules/.bin/eslint --init

# 输入上面的命令后，终端会让你填写一份问卷，根据填写情况来生成你的 eslintrc.js

# 你可以根据你的需求填写，也可以参考一下我的
# 👇👇👇
# How would you like to use ESLint ? To check syntax, find problems, and enforce code style

# What type of modules does your project use ? JavaScript modules (import/export)

# which framework does your project use ? React

# Does your project use TypeScript ? No

# Where does your code run ? Browser, Node

# How would you like to define a style for your project ? Answer questions about your style

# what format do you want your config file to be in ? JavaScript

# what style of indentation do you use ? Tabs

# what quotes do you use for strings ? Single

# what line endings do you use ? Windows

# Do you require semicolons ？Yes

// 生成的eslintrc.js

module.exports = {
  env: {
    browser: true,
    es2021: true,
    node: true,
  },
  extends: ["eslint:recommended", "plugin:react/recommended"],
  parserOptions: {
    ecmaFeatures: {
      jsx: true,
    },
    ecmaVersion: 12,
    sourceType: "module",
  },
  plugins: ["react"],
  rules: {
    indent: ["error", "tab"],
    "linebreak-style": ["error", "windows"],
    quotes: ["error", "single"],
    semi: ["error", "always"],
  },
};

Preitter

安装Prettier，Husky，Lint-Staged三个库

⚠ cnpm是配置了淘宝源的npm，没配置就用npm或者yarn

⚠ 一定要加 --save-dev，确保这三个依赖是开发环境的依赖

# 规范代码风格
cnpm install --save-dev --save-exact prettier
# 让开发者更方便的设置git hook
cnpm install --save-dev husky
# 在提交代码时通过 ESLint 和 Prettier 依次检查有改动的代码，卡住不合规范的代码
cnpm install --save-dev lint-staged

在你的根目录下建立 .prettierrc.js 进行初始化配置

✨更多配置在这儿

module.exports = {
  trailingComma: "all", // 函数最后不需要逗号
  printWidth: 80, // 超过最大值换行
  tabWidth: 2, // 缩进字节数
  semi: true, // 格式化在句尾添加分号
  singleQuote: true, // 使用单引号替换双引号
  jsxBracketSameLine: true, // 把 > 单独放一行
  jsxSingleQuote: true, // 在jsx中使用单引号替换双引号
  arrowParens: "always", // 箭头函数参数只有一个是否要小括号
};

Lint-Staged

初始化Lint-Staged的配置

npx mrm lint-staged

⚠ 可能会报错 Cannot find module ‘ …省…略…\node_modules\mrm-task-lint-staged’

# 降低mrm版本
npx mrm@2 lint-staged

执行完成后会在你的package.json中生成husky和lint-staged的相关配置，你也可以进行手动配置

  "husky": {
    "hooks": {
      "pre-commit": "npm run lint-staged"
    }
  },
  "lint-staged": {
    "**/*.{js,jsx,ts,tsx}": "npm run lint-staged:js",
    "**/*.{js,jsx,tsx,ts,less,md,json}": [
      "prettier --write",
      "git add"
    ]
  }

最后配置一下 scripts 中的脚本命令

"lint": "npm run lint:js && npm run lint:style && npm run lint:prettier",
"lint-staged": "lint-staged",
"lint-staged:js": "eslint --ext .js,.jsx,.ts,.tsx ",
"lint:fix": "eslint --fix --cache --ext .js,.jsx,.ts,.tsx --format=pretty ./src && npm run lint:style",
"lint:js": "eslint --cache --ext .js,.jsx,.ts,.tsx --format=pretty ./src",
"lint:prettier": "check-prettier lint",
"lint:style": "stylelint --fix \"src/**/*.less\" --syntax less",
"prettier": "prettier -c --write \"**/*\""

大功告成 👌👌👌 现在来尝试改动一些东西然后推送到你的远端仓库，康康是什么效果 🕵️‍♂️

总结

工具虽然有作用，但上限也就仅仅是解决工程上的问题，工程的问题一定会有工程的方法来解决。真正降低开发效率的还是人写的生涩难懂、逻辑混乱的Bad Code，如何去提升自己代码的可读性，可复用性，可重构性才是大家要去思考的问题。
