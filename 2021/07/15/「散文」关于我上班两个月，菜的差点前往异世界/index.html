关于我上班两个月，菜的差点直接转生到异世界

前情提要

来柚子鸭刚好两个月出头，这次迭代是我参与的第四次迭代。这篇文章也是围绕这次迭代，本迭代我的第一个需求的大致背景是对一个表单页中的可编辑表格进行改造，添加一列上传图片的组件。完成这个需求和解决相关缺陷的过程反映这段时间我的成长和目前自身存在的问题，抽个时间来总结总结。

下面由我本色出演！🙋‍♂

环境

  • 框架： React
  • 组件： Antd， Antd Pro

见招拆招 🤡

功能点详情: 目前需要改造的表单页中有一个可编辑表格，可编辑表格可分为编辑态和展示态，默认为展示态。展示态时，行末尾有编辑和删除两个按钮，点击编辑进入编辑态可以编辑当行单元格内的数据；点击删除可以删除当行的数据。编辑态时，行末尾有保存和取消两个按钮，点击保存，若当行所有单元格数据符合自己的校验规则，则保存成功，否则失败；点击取消则直接放弃当前正在编辑的数据。表格最下面有一个创建新行的按钮，当表格中没有正在编辑的行时能成功创建，若有则需要取消当行或者保存当行后才能创建。

需求：新加一列，编辑态时上传图片，展示态时展示图片。

Upload 组件不能正常加载已有图片

缺陷背景： 当编辑表格为展示态时，Upload 单元格内展示上传的图片和图片名称（带后缀），二者所在轴为纵轴；为编辑态时展示 Upload 组件和已经上传的文件（可为空），二者所在轴为纵轴。对已存在行点击编辑进入编辑态，只展示了上传按钮，没有展示当前行的图片。

问题分析：点击编辑时，向后端请求时数据就没到位，或者流入表格组件的数据有图片数据但是 Upload 组件没有接收到。

解决方案：先检查从后端请求过来的数据，发现数据其实是被请求过来的了。那做法就简单了单走一个 useState，当点击编辑的时候把那行的数据设置个那个 state 然后将其作为参数传给表格组件。再去表格组件内打印，有数据了，但是 Upload 组件那个已有图片的样式还是没加载出来。那结论只有一个了，传过来的数据结构和 Upload 接收的数据结构没对上。去 antd 官网查之后发现 Upload 接收的数据为如下的格式：

[
    {
        url: '....',
        uid: '....',
        name: '....',
    },
    ....
]

而我自己传过来的图片数据就是一个对象而已，对数据格式化一下，再进去 OK 了，一切都在我 JOJO 的计算之中 🤡。

新建行上传图片后选择旁边的 Select 组件会被刷新掉

缺陷背景：所有新建的行，若先上传了图片，点击旁边的 Select 组件会刷新掉已上传的图片。

问题分析：我编辑态展示的已有图片是取点击编辑时保存到 state 的数据，那我新建的时候 state 里压根就没有数据，档然不显示喽。

解决方案： 这里使用的 Upload 组件是嵌在一个表单组件上的，表单组件提供了一个表单实例，上面有一个方法 getFieldValue 可以通过组件的 name 来获取表单的值，那我直接上传了就获取一遍放在 state 里，然后再把 state 的值给 Upload 的 file 属性不就完事了吗？展示的时候优先取新建的值，其次取当前值，如果都没有就设置为空。这波是受控组件，实在是高 🤡。

点击编辑后，点击取消然后新建一行会把上一次点击编辑那行的图片带下来

缺陷背景：对已有行点击编辑，然后点击取消，然后新建一行，新建的那行会把上一次点击那行的图片带下来展示。

问题分析：一看到这个问题我就明白了，清空编辑时保存的 state 的操作我只在保存的末尾设置了，取消/新建的时候没有设置，那此时新建行使用的数据是上一次点击编辑的存的 state。

解决方案：在取消/新建的时候设置清空 state 的操作，这波啊还是受控组件，高实在是高。🤡

新建/编辑行后未进行保存/取消行，能成功保存整个表单

缺陷背景：点击新建或者编辑后处于编辑态，直接提交整个表单能成功提交。

问题分析：历史遗留问题，另一个页面做了处理，参照着来就行了。

解决方案：设置一个状态值默认为-1，新建编辑时就设置为表格总共的行数，保存和取消时就设置回-1，antd 的 Form 组件有一个 rules 的属性，在里面自定义规则 validator 判断大于-1 时就抛出一个 reject 的 Promise。

尘归尘，土归土，🥦🐕 还是 🥦🐕

如果你已经看到这儿，相信已经发现前面解决的缺陷除了最后一个问题其他多多少少都是有关联的，而且没有规避一个致命的问题————为什么点击编辑的时候其他数据都有就图片没有？💢

是的，数据并非没有，而是单纯的没有设置进去 🤡。为什么要说没有设置进去，是因为我在检查组件代码的时候发现其实组件内部已经设置了一个 state 来管理每一行的数据了。console.log一下图片的数据赫然出现在控制台中，是一个对象。再更深入的进入 Upload 组件，打印它的 props，props 中 file 属性是 undefined，但是 value 属性有对应的值了。兴致勃勃的改造完 Upload 组件，又回到了第三个缺陷 🙇‍♂️，平时传值都是通过 file 属性传递，现在多了个 value，得考虑清空缓存的一个时机。越做越复杂，直接血压拉满 💢💢。

最后想不出很好的解决方案了，去请教了下 leader，leader 果然是 leader🙏，一看就发现了我少传了一个属性valuePropName="file"，这个属性最终是直接设置给 input 标签的。一般来说表单控件值的属性名都是 value，所以直接设置就行了，但是 Upload 组件最终反馈到 HTML 上就是一个 upload 类型的 input，它接收的属性名不是 value 而是 file，所以 Upload 组件的 value 有值而 file 为 undefined 就解释的通了。补传这个属性后，一切都变得如德芙一样丝滑。

总结一下，在思考如何解决这一系列缺陷的时候，我是凭着这两个月对公司组件库的熟悉程度，以及这段时间所接触到的概念和方法（比如受控组件的概念，React 组件通信的一些技巧）来指定解决方案的，但最终千言万语最终只换得一句话 😭。成长是有的，但归根结底还是基础太差，而且是自身知识系统上的漏洞，如果 leader 不说我压根都不知道有valuePropsName这个属性。目前我认为比较好的解决方案还是踩一个坑记一个坑，之后碰见相关问题能够有端联想 🙇‍♂️。
